02 String Functions

2.1 UPPER( ) and LOWER( )

UPPER() converts text to uppercase letters. LOWER() converts text to lowercase letters.

SQL 
SELECT UPPER('hello world') -- Result: HELLO WORLD
SELECT LOWER('HELLO WORLD') -- Result: hello world
SELECT UPPER(first_name), LOWER(last_name) FROM employees

2.2 LEN( ) and DATALENGTH( )

Both functions count the number of characters in a text. The key difference is how they handle spaces:

  • LEN() — ignores trailing spaces but counts spaces at the beginning.
  • DATALENGTH() — counts all characters including trailing spaces (and returns bytes, not just characters).
SQL 
SELECT LEN('Hello   ')        -- Result: 5 (trailing spaces ignored)
SELECT DATALENGTH('Hello   ') -- Result: 8 (all bytes counted)

2.3 LEFT( ) and RIGHT( )

LEFT(column, n) extracts the first n characters from the start of a string. RIGHT(column, n) extracts the last n characters from the end. Here, n means number of characters, not position.

SQL 
SELECT LEFT(category, 3)  AS first_3_chars FROM gold.dim_products
SELECT RIGHT(phone, 4)    AS last_4_digits  FROM customers
ℹ️
Behavior Note

If n is greater than the string length, LEFT() and RIGHT() simply return the full string — no error is thrown.

2.4 SUBSTRING( )

SUBSTRING(column, start_position, length) extracts a portion of characters from any position in a text. It is the most flexible string extraction function and can replicate both LEFT and RIGHT behavior. Position starts at 1, not 0.

🎯 Task

In a column, data is stored as (FF-Name). Extract only the name information using SUBSTRING.

SQL 
SELECT
    SUBSTRING(info, 4, LEN(TRIM(info))) AS name  -- Extract from 4th position to last
FROM customers

SUBSTRING can also replace LEFT and RIGHT:

Replacement of LEFT: SUBSTRING(column, 1, ending_position)
Replacement of RIGHT: SUBSTRING(column, LEN(column)-2, LEN(column))
🎯 Task

Extract the first and last 3 characters from the category column of the products table using different string functions.

SQL 
SELECT
    LEFT(category, 3)                                        AS leftinfo,
    SUBSTRING(category, 1, 3)                                 AS subinfo1,
    RIGHT(category, 3)                                       AS rightinfo,
    SUBSTRING(category, LEN(TRIM(category))-2, LEN(Category)) AS subinfo2
FROM gold.dim_products
WHERE Category IS NOT NULL

2.5 CHARINDEX( ) and PATINDEX( )

CHARINDEX (Character Index) finds the position of one or multiple characters together inside a string. PATINDEX (Pattern Index) finds the position of a pattern using wildcard characters.

🎯 Task

Find the position of 'r' and 'Ra' in the first name column of employees.

SQL 
SELECT
    firstname,
    CHARINDEX('r', FirstName)   AS r_position,
    CHARINDEX('Ra', FirstName)  AS Ra_position,
    PATINDEX('%r%', FirstName) AS r_pattern_position
FROM sales.employees

2.6 REPLACE( )

Syntax: REPLACE(original_string, old_substring, new_substring) — replaces part of a text.

🎯 Task — Remove unwanted characters

Remove '-' from phone numbers.

SQL — Clean Phone Numbers 
SELECT REPLACE(phone_number, '-', '') AS clean_number
FROM customers;
-- Output: 017-123-4567 → 0171234567
🎯 Task — Replace multiple words

Convert the words 'bad' and 'poor' to 'good' and 'excellent' respectively.

SQL — Nested REPLACE 
SELECT REPLACE(REPLACE(description, 'bad', 'good'), 'poor', 'excellent')
FROM reviews;
🎯 Task — Mask Sensitive Data

Hide the first 12 digits of a card number with '*'.

SQL — Data Masking 
SELECT REPLACE(card_number, LEFT(card_number, 12), '************')
FROM payments;
-- Output: 1234567890123456 → ************3456

2.7 TRIM( ), LTRIM( ), RTRIM( )

All three functions remove unwanted spaces (or specified characters) from text.

  • LTRIM() — removes spaces from the left (beginning) of the string.
  • RTRIM() — removes spaces from the right (end) of the string.
  • TRIM() — removes spaces from both sides. Can also remove specific characters.
SQL — Special TRIM Case 
SELECT TRIM('#' FROM '###Data###')
-- Result: 'Data'
💡
Pro Tip

Use TRIM() to remove specific characters (not just spaces) using the FROM syntax — a powerful but lesser-known feature in SQL Server 2017+.

03 Number Functions

3.1 ROUND( ), CEILING( ), FLOOR( )

These are the core rounding functions in SQL. Syntax: ROUND(number, decimal_places)

ℹ️
Negative Decimal Places

Using a negative number for decimal places rounds to the left of the decimal point. For example, ROUND(234.545, -1) rounds to the nearest ten → 230.

🎯 Task

Practice rounding a number at different decimal places — including positive and negative values.

SQL — ROUND Practice 
SELECT ROUND(234.545, 2)  AS RoundedValue UNION ALL  -- 234.550
SELECT ROUND(234.545, 1)             UNION ALL  -- 234.500
SELECT ROUND(234.545, 0)             UNION ALL  -- 235.000
SELECT ROUND(234.545, -1)            UNION ALL  -- 230.000
SELECT ROUND(-234.545, 2)            UNION ALL  -- -234.550
SELECT ROUND(-234.545, 1)                         -- -234.500

CEILING always rounds a number up to the nearest integer, regardless of the decimal fraction. It behaves toward positive infinity.

SQL — CEILING 
SELECT CEILING(123.456)   -- Result: 124
SELECT CEILING(-123.456)  -- Result: -123 (moves toward zero = up toward +∞)

FLOOR always rounds a number down to the nearest integer, regardless of the decimal fraction. It behaves toward negative infinity.

SQL — FLOOR 
SELECT FLOOR(123.456)   -- Result: 123
SELECT FLOOR(-123.456)  -- Result: -124 (moves away from zero = down toward -∞)

3.2 ABS( )

ABS(number) returns the absolute (positive) value of a number, regardless of whether it was positive or negative.

SQL 
SELECT ABS(-450)  -- Result: 450
SELECT ABS(450)   -- Result: 450

04 Date and Time Functions — Part Extraction

DAY MONTH YEAR DATEPART DATENAME DATETRUNC EOMONTH

4.1 DAY( ), MONTH( ), YEAR( )

These are simple shortcut functions to extract a specific part of a date:

  • DAY(date_column) — extracts the day number from the date.
  • MONTH(date_column) — extracts only the month number.
  • YEAR(date_column) — extracts only the year.
🎯 Task

Find how many orders were placed in each year.

SQL 
SELECT
    YEAR(order_date) AS order_year,
    COUNT(*)        AS total_orders
FROM gold.fact_sales
WHERE order_date IS NOT NULL
GROUP BY YEAR(order_date)
ORDER BY order_year

4.2 DATEPART( )

DATEPART(part, date_column) accepts two parameters: the part (Day, Week, Month, Quarter, Year, Hour, etc.) and the column name. You can use abbreviations instead of full names — for example, DATEPART(MM, column_name) for month.

4.3 DATENAME( )

DATENAME(part, date_column) provides the name of the extracted part. For example, it returns 'August' for a month or 'Wednesday' for a weekday.

⚠️
Important Distinction

DATENAME(day, column) returns the day number, not the day name. To get the day name, use DATENAME(WEEKDAY, column). Also, DATENAME results are stored as characters, not integers — so use DATEPART if you need numeric values for calculations.

4.4 DATETRUNC( )

DATETRUNC(part, date) truncates a datetime value to the specified part. Everything after that part is reset to its minimum value (zero or 1).

SQL — DATETRUNC Example 
-- Input: '2026-04-28 13:08:10.117'
SELECT DATETRUNC(Minute, GETDATE()) AS current_date_time
-- Result: 2026-04-28 13:08:00.000 (seconds & ms reset to 0)

4.5 EOMONTH( )

EOMONTH(date_column) replaces the day information with the last day of that month. Very useful for financial dashboards and monthly period filtering.

05 Date Manipulation: DATEADD & DATEDIFF

5.1 DATEADD( )

DATEADD(part, number, date) adds or subtracts a specified amount from a date part. Use a negative number to go backward in time.

🎯 Task

Find the date value from 7 days before the current date.

SQL — DATEADD 
SELECT DATEADD(day, -7, GETDATE()) AS previous_week           -- Returns datetime
SELECT CAST(DATEADD(day, -7, GETDATE()) AS DATE) AS casted_previous_week  -- Date only
🎯 Task

Get all order details from the last 7 days.

SQL 
SELECT * FROM sales.orders
WHERE Orderdate >= DATEADD(DAY, -7, GETDATE())

5.2 DATEDIFF( )

DATEDIFF(part, start_date, end_date) calculates the difference between two dates in the specified unit.

🎯 Task

Calculate the difference in days between order date and ship date.

SQL — DATEDIFF 
SELECT
    sls_order_dt,
    sls_ship_dt,
    DATEDIFF(Day, sls_order_dt, sls_ship_dt) AS daytoship
FROM silver.crm_sales_details
⚠️
DATEDIFF Boundary Trap

DATEDIFF(DAY, '2026-05-01 23:59', '2026-05-02 00:01') returns 1 — only the date boundary crossing matters, not the actual time difference (2 minutes). Use seconds for precision: DATEDIFF(SECOND, start, end) / 3600.0. The .0 is critical to get a decimal result.

06 FORMAT( ) Function

FORMAT(value, format [, culture]) returns a value formatted with the specified format and optional culture. It is primarily used for formatting dates, times, and numbers into human-readable strings.

ℹ️
Parameters

value = the value to format. format = format string (standard or custom). culture = optional locale identifier (e.g., 'en-US', 'fr-FR').

6.1 Formatting Numbers

SQL — Number Formatting 
SELECT FORMAT(1234567, '##,##,##,###')  -- Result: 12,34,567

SELECT
    FORMAT(1234.56, 'C') AS Currency,    -- $1,234.56
    FORMAT(1234.56, 'N') AS Number,      -- 1,234.56
    FORMAT(0.756, 'P')   AS Percentage;  -- 75.60%

6.2 Formatting Dates

SQL — Date Formatting 
SELECT
    FORMAT(GETDATE(), 'yyyy-MM-dd')  AS formatted_date,
    FORMAT(GETDATE(), 'dd/MM/yyyy')  AS British_Format,
    FORMAT(GETDATE(), 'MMMM dd, yyyy') AS Full_Format,
    FORMAT(GETDATE(), 'ddd')          AS Day_Short,
    FORMAT(GETDATE(), 'dddd')         AS Day_Full

6.3 Culture / Localization

SQL — Culture Support 
SELECT
    FORMAT(1234.56, 'C', 'en-US') AS US_Format,     -- $1,234.56
    FORMAT(1234.56, 'C', 'fr-FR') AS French_Format  -- 1 234,56 €
💎
International Reports

Culture-aware formatting is very useful when working with international reports — you can display currency, dates, and numbers in the local format for each region without changing the underlying data.

6.4 Custom Date & Time Patterns

07 CONVERT( ) Function

CONVERT(data_type, value [, style]) converts a value to a different data type and also formats the value. It is SQL Server specific.

SQL — CONVERT Examples 
SELECT CONVERT(INT, '125')                   -- String '125' → Integer 125
SELECT CONVERT(VARCHAR, GETDATE(), 103)      -- Output: 02/05/2026 (dd/mm/yyyy)

SELECT
    CONVERT(VARCHAR, OrderDate, 120) AS FormattedDate
FROM Orders;

💎
When to Use CONVERT

CONVERT is faster than FORMAT() for date formatting in queries. Use it when performance matters — especially in large datasets, joins, or WHERE clause filtering. However, it has limited style codes compared to FORMAT's full flexibility.

08 CAST( ) Function

CAST(value AS data_type) converts a value into a specified data type. It is ANSI SQL standard, making it the most portable conversion function across different database platforms.

SQL — CAST Examples 
SELECT CAST('2026-05-01' AS DATE)         -- String → Date
SELECT CAST(123.45 AS INT)               -- Float → Integer (123)
SELECT CAST(DATEADD(day, -7, GETDATE()) AS DATE) AS last_week
💡
TRY_CAST and TRY_CONVERT

For dirty or untrusted data (e.g., ETL pipelines), use TRY_CAST() and TRY_CONVERT() instead. They return NULL instead of throwing an error if the conversion fails.

SQL — Safe Conversion 
TRY_CAST('abc' AS INT)     -- Result: NULL (no error thrown)
TRY_CONVERT(INT, 'xyz')   -- Result: NULL (no error thrown)

Case Sensitivity in FORMAT

10 Pattern Matching: LIKE, %, _

Used to search for a pattern inside a column. Pattern matching is done with the LIKE operator combined with wildcard characters.

% → any number of characters (0, 1, or many)
_ → exactly one character
SQL — LIKE Pattern Examples 
SELECT * FROM customers WHERE name LIKE 'A%'      -- Starts with 'A'
SELECT * FROM customers WHERE name LIKE '%son'    -- Ends with 'son'
SELECT * FROM customers WHERE name LIKE '%ar%'    -- Contains 'ar' anywhere
SELECT * FROM customers WHERE code LIKE 'A_C'     -- A, any 1 char, then C (e.g., 'ABC')
❌
Performance Warning

Using LIKE '%value%' with a leading wildcard forces a full table scan and cannot use an index. Avoid leading wildcards on large tables. Use full-text search or indexed views for large-scale pattern matching.

11 Pro Tips

⚡ Performance & Best Practice Tips

1

Avoid FORMAT on large datasets. It is a performance killer. Use FORMAT only for reporting and final display layers. Never use it in core queries, aggregations, or joins.

2

Calculate day differences using seconds. When data is in datetime format, use DATEDIFF(SECOND, start, end) / 3600.0. The .0 is crucial — without it, you get only integer results.

3

Use DATETRUNC instead of GROUP BY DATEPART for cleaner grouping. DATETRUNC produces proper date values that sort and display correctly in reports.

4

Use EOMONTH for financial dashboards. It instantly gives you the last day of any month, which is essential for period-end reporting and financial summaries.

5

Always use DATEADD in filters for relative date ranges (e.g., last 7 days, last 30 days). It is dynamic and handles month-end boundaries correctly.

6

Never use functions on indexed columns in WHERE clause. Index gets deactivated. Rewrite to keep the column bare on one side of the comparison.

7

Always use ISO Date format (2026-04-25) in queries. It is unambiguous across all SQL Server language settings and avoids locale-related date parsing errors.

8

Use EOMONTH trick for monthly filtering. Combine DATETRUNC and DATEADD to create safe, index-friendly monthly filters.

9

Use DATEFROMPARTS() for safe date creation. Risky: '2026-13-01' may pass as string. Safe: SELECT DATEFROMPARTS(2026, 5, 2) — SQL validates the values.

10

Use TRY_CAST / TRY_CONVERT for dirty data. Essential for ETL pipelines. TRY_CAST('anc' AS INT) returns NULL gracefully instead of crashing your pipeline.

11

Avoid repeated function calls. Store the result in a variable for reuse: DECLARE @Last7Days DATE = DATEADD(DAY, -7, GETDATE());. This improves readability and slight performance.

SQL — EOMONTH Monthly Filter Pattern 
WHERE OrderDate >= DATETRUNC(MONTH, Orderdate)
  AND OrderDate <  DATEADD(MONTH, 1, DATETRUNC(MONTH, Orderdate))

12 Traps and Solutions

Trap 1 — Functions in WHERE Clause Deactivate Indexes

Wrapping an indexed column inside a function in the WHERE clause prevents SQL Server from using the index, causing a full table scan.

SQL — Index Trap vs Fix 
-- ❌ BAD — CAST wraps indexed column, forces full scan
SELECT * FROM Orders
WHERE CAST(OrderDate AS DATE) = '2026-05-02';

-- ✅ GOOD — Column is bare, index is used
SELECT * FROM Orders
WHERE OrderDate >= '2026-05-02'
  AND OrderDate <  '2026-05-03';

Trap 2 — DATEDIFF Boundary vs Actual Time Difference

DATEDIFF counts date boundary crossings, not actual elapsed time. This causes subtle bugs when working with datetime values near midnight.

SQL — DATEDIFF Boundary Trap 
SELECT DATEDIFF(DAY, '2026-05-01 23:59', '2026-05-02 00:01')
-- Output: 1 (only the date boundary crossed matters — actual gap is 2 minutes!)

-- ✅ Use seconds for precision:
SELECT DATEDIFF(SECOND, start_date, end_date) / 3600.0 AS hours_elapsed
-- 3600.0 is crucial — without it, result is truncated to integer

Trap 3 — LIKE with Leading Wildcard

A pattern with a leading % cannot use an index and forces a full table scan. Especially harmful on large production tables.

SQL — LIKE Wildcard Trap 
-- ❌ Slow — leading wildcard forces full table scan
SELECT * FROM customers WHERE name LIKE '%Smith';

-- ✅ Fast — trailing wildcard can still use an index
SELECT * FROM customers WHERE name LIKE 'Smith%';
💎
Summary: The Golden Rules

Think of FORMAT as a presentation layer tool — not a data processing tool. Keep indexed columns bare in WHERE/JOIN conditions. Use TRY_CAST/TRY_CONVERT for ETL. Always use ISO dates. Precision matters in datetime calculations — always work in seconds, then convert.
