01What Is a Subquery?

A subquery (also called an inner query or nested query) is a SELECT statement embedded inside another SQL statement. The outer query is called the main query or outer query. The subquery executes first, and its result is passed to the outer query for further processing.

ℹ️
Simple Definition

A subquery is a query inside a query. Think of it as asking SQL: "First go find me this set of data, then use that answer to solve my bigger question."

Basic Syntax

SELECT column1, column2 FROM table_name WHERE column1 = ( SELECT column1 FROM another_table WHERE condition );

Sample Tables Used Throughout This Guide

T-SQL — Setup
-- Employees table
CREATE TABLE Employees (
    EmpID      INT PRIMARY KEY,
    EmpName    VARCHAR(100),
    DeptID     INT,
    Salary     DECIMAL(10,2),
    ManagerID  INT,
    HireDate   DATE
);

-- Departments table
CREATE TABLE Departments (
    DeptID    INT PRIMARY KEY,
    DeptName  VARCHAR(100),
    Budget    DECIMAL(12,2)
);

-- Orders table
CREATE TABLE Orders (
    OrderID     INT PRIMARY KEY,
    CustomerID  INT,
    ProductID   INT,
    Quantity    INT,
    Amount      DECIMAL(10,2),

);

-- Customers table
CREATE TABLE Customers (
    CustomerID    INT PRIMARY KEY,
    CustomerName  VARCHAR(150),

);

-- Products table
CREATE TABLE Products (
    ProductID    INT PRIMARY KEY,
    ProductName  VARCHAR(150),
    Category     VARCHAR(100),
    UnitPrice    DECIMAL(10,2),
    Stock        INT
);

02Why Use Subqueries?

Subqueries solve problems that cannot be easily solved with a single flat SELECT statement. They let you break a complex question into smaller, logical steps.

🔍 Filter with computed values

Find employees who earn more than the average salary — the average itself must be calculated first.

🚫 Negative existence checks

Find customers who have never placed an order — requires checking absence in another table.

📊 Aggregated comparisons

Compare each department's salary against the company-wide average using a subquery in SELECT.

🏗 Multi-step logic

First rank products by sales, then filter the top 5 — two logical steps, one query.

📐 Reusable result sets

Treat a subquery like a temporary virtual table (derived table) to simplify complex joins.

✏️ DML operations

UPDATE or DELETE rows that meet criteria defined by another query — subqueries inside DML.

03How Subqueries Work Internally

Understanding what SQL Server does behind the scenes with subqueries is key to writing efficient queries and interpreting execution plans.

The Internal Processing Steps

  1. Parse: SQL Server parses the entire statement, identifying the inner and outer queries.
  2. Bind: The query optimizer resolves all object names and column references.
  3. Optimize: The optimizer may unnest the subquery and convert it to an equivalent JOIN, or it may keep it as-is if that is more efficient.
  4. Execute: For non-correlated subqueries, the inner query runs once. For correlated subqueries, it may run once per row of the outer query.
  5. Return: The result of the inner query (a scalar value, a list, or a table) is passed back to the outer query.
💡
Optimizer Unnesting

SQL Server's query optimizer is very smart. It often converts subqueries into JOIN operations automatically because joins are generally easier to optimize. This is why a subquery and an equivalent JOIN often produce identical execution plans.

Non-Correlated vs. Correlated Execution

04Execution Flow of Subqueries

Visualizing the execution flow helps you reason about your query's correctness and performance.

Non-Correlated Flow

T-SQL
-- Step 1: Inner query runs ONCE → returns 6000.00
-- Step 2: Outer query filters using that value

SELECT EmpName, Salary
FROM   Employees
WHERE  Salary > (SELECT AVG(Salary) FROM Employees);

-- FLOW:
--   [Inner]  SELECT AVG(Salary) FROM Employees → 6000.00
--   [Outer]  SELECT EmpName, Salary FROM Employees WHERE Salary > 6000.00

Correlated Flow

T-SQL
-- For EACH row of the outer query, the inner query re-executes

SELECT e.EmpName, e.Salary, e.DeptID
FROM   Employees e
WHERE  e.Salary > (
           SELECT AVG(e2.Salary)
           FROM   Employees e2
           WHERE  e2.DeptID = e.DeptID  -- references outer alias 'e'
       );

-- FLOW (for 1000 rows in outer query):
--   Row 1 (DeptID=10): Inner runs → AVG salary for Dept 10 = 5500
--   Row 2 (DeptID=10): Inner runs → AVG salary for Dept 10 = 5500
--   Row 3 (DeptID=20): Inner runs → AVG salary for Dept 20 = 7200
--   ... repeats 1000 times total
⚠️
Performance Warning

A correlated subquery executing once per outer row is called a row-by-row (RBAR) operation. On a table with 1 million rows, the inner query might execute 1 million times. Always look for alternatives using JOINs or window functions.

06Single-Row Subquery

A single-row subquery returns exactly one row and one column. It must be used with single-value comparison operators: =, <, >, <=, >=, <>.

🚨
Runtime Error Risk

If a single-row subquery returns more than one row, SQL Server throws: Subquery returned more than 1 value. This is not permitted... Always ensure the inner query is guaranteed to return at most one row.

Example 1 — Find employees earning the maximum salary

T-SQL
SELECT EmpName, Salary
FROM   Employees
WHERE  Salary = (SELECT MAX(Salary) FROM Employees);

-- Inner query: SELECT MAX(Salary) FROM Employees → 95000.00
-- Outer query: WHERE Salary = 95000.00
-- Output: All employees who earn exactly 95000.00

Example 2 — Find departments with above-average budget

T-SQL
SELECT DeptName, Budget
FROM   Departments
WHERE  Budget > (SELECT AVG(Budget) FROM Departments);

-- Inner: AVG(Budget) → 500000.00
-- Returns departments whose budget exceeds the overall average

07Multiple-Row Subquery

A multiple-row subquery returns more than one row (but typically one column). It must be used with multi-value operators: IN, NOT IN, ANY, ALL, EXISTS, NOT EXISTS.

Example — Find employees in the IT or Finance department

T-SQL
SELECT EmpName, DeptID
FROM   Employees
WHERE  DeptID IN (
           SELECT DeptID
           FROM   Departments
           WHERE  DeptName IN ('IT', 'Finance')
       );

-- Inner: Returns {3, 5} (DeptIDs for IT and Finance)
-- Outer: WHERE DeptID IN (3, 5)
-- Result: All employees in those two departments
⚠️
Do NOT use = with a multi-row subquery

Using = (subquery) when the subquery can return multiple rows will cause a runtime error. Use IN instead when multiple rows are possible.

08Multiple-Column Subquery

A multiple-column subquery returns multiple columns. It is commonly used in FROM (as a derived table) or with EXISTS for composite-key matching.

Example — Match on composite key (CustomerID + ProductID)

T-SQL
SELECT o.OrderID, o.CustomerID, o.ProductID, o.Amount
FROM   Orders o
WHERE  EXISTS (
           SELECT 1
           FROM   PriorityCustomers pc
           WHERE  pc.CustomerID = o.CustomerID
             AND  pc.ProductID  = o.ProductID
       );

-- Finds orders where both CustomerID and ProductID
-- match a record in the PriorityCustomers table

09Correlated Subquery

A correlated subquery references one or more columns from the outer query. This creates a dependency: the inner query cannot run independently. It re-evaluates for each row of the outer query.

Business Example — Employees earning above their department average

T-SQL
SELECT e.EmpName, e.Salary, e.DeptID
FROM   Employees e
WHERE  e.Salary > (
           SELECT AVG(e2.Salary)
           FROM   Employees e2
           WHERE  e2.DeptID = e.DeptID  -- correlated reference
       );

-- For each employee (outer row), the inner query calculates
-- the average salary for THAT specific employee's department.
-- Result: Employees whose personal salary > their dept average

Line-by-Line Output Explanation

EmpNameSalaryDeptIDDept AvgIncluded?
Alice90,0001070,000✔ Yes
Bob65,0001070,000✘ No
Carol80,0002075,000✔ Yes
Dave60,0002075,000✘ No
💎
Better Alternative with Window Functions

For large datasets, replace correlated subqueries with window functions. Use AVG(Salary) OVER (PARTITION BY DeptID) — it scans the table once instead of once per row, dramatically improving performance.

10Nested Subquery

A nested subquery is a subquery inside another subquery — multiple levels of nesting. SQL Server supports deep nesting (up to 32 levels), though deep nesting usually signals a design that needs rethinking.

Example — Three levels of nesting

T-SQL
-- Level 1 (outermost): Find employees
-- Level 2: In departments with above-average budget
-- Level 3: Where average budget is computed from all departments

SELECT EmpName, DeptID
FROM   Employees
WHERE  DeptID IN (
           SELECT DeptID
           FROM   Departments
           WHERE  Budget > (       -- Level 2 subquery
               SELECT AVG(Budget)
               FROM   Departments  -- Level 3 subquery
           )
       );
💡
Readability Tip

When you find yourself nesting more than 2 levels deep, consider rewriting using CTEs. CTEs give each logical step a name, making the query far easier to read, debug, and maintain.

11Scalar Subquery

A scalar subquery returns exactly one value (one row, one column). It can appear anywhere a single value is expected — in the SELECT list, WHERE, HAVING, or even ORDER BY.

Example — Show each employee's salary and company average side by side

T-SQL
SELECT
    EmpName,
    Salary,
    (SELECT AVG(Salary) FROM Employees)    AS CompanyAvgSalary,
    Salary - (SELECT AVG(Salary) FROM Employees) AS DiffFromAvg
FROM   Employees;

-- Output sample:
-- EmpName  | Salary   | CompanyAvgSalary | DiffFromAvg
-- Alice    | 90000    | 70000            | +20000
-- Bob      | 65000    | 70000            | -5000

12Subquery with EXISTS & NOT EXISTS

EXISTS checks whether a subquery returns any rows at all. It returns TRUE as soon as the first matching row is found and stops — this is called a short-circuit evaluation, making it very efficient.

EXISTS — Customers who have placed at least one order

T-SQL
SELECT c.CustomerName, c.Email
FROM   Customers c
WHERE  EXISTS (
           SELECT 1
           FROM   Orders o
           WHERE  o.CustomerID = c.CustomerID
       );

-- SELECT 1 is a convention — we only care if any row exists,
-- not what the row contains. Using 1 is slightly more readable
-- than SELECT * and signals intent clearly.

NOT EXISTS — Customers who have NEVER ordered (churn detection)

T-SQL
SELECT c.CustomerName, c.Email, c.City
FROM   Customers c
WHERE  NOT EXISTS (
           SELECT 1
           FROM   Orders o
           WHERE  o.CustomerID = c.CustomerID
       );

-- Business use: Email marketing team uses this list
-- to send "We miss you!" re-engagement campaigns.
💎
EXISTS vs IN — Which is Faster?

EXISTS short-circuits on the first match — ideal when you expect many matches. IN builds the full list first. For large datasets with NULLs in the list, NOT EXISTS is safer and often faster than NOT IN, because NOT IN with any NULL in the subquery returns zero rows.

13Subquery with IN & NOT IN

IN checks if a value matches any value in the subquery's result set. NOT IN checks that a value does not appear in the result set.

IN — Products ordered in January 2024

T-SQL
SELECT ProductName, Category, UnitPrice
FROM   Products
WHERE  ProductID IN (
           SELECT DISTINCT ProductID
           FROM   Orders
           WHERE  OrderDate >= '2024-01-01'
             AND  OrderDate < '2024-02-01'
       );

NOT IN — Products that have NEVER been ordered

T-SQL
SELECT ProductName, Stock
FROM   Products
WHERE  ProductID NOT IN (
           SELECT DISTINCT ProductID
           FROM   Orders
           WHERE  ProductID IS NOT NULL  -- CRITICAL: exclude NULLs
       );
🚨
NULL Trap with NOT IN

If the subquery inside NOT IN returns even a single NULL, the entire NOT IN condition evaluates to UNKNOWN and returns zero rows — silently! Always add WHERE column IS NOT NULL inside the subquery, or use NOT EXISTS instead.

14Subquery with ANY & ALL

ANY (synonym: SOME) returns TRUE if the comparison is TRUE for at least one value in the subquery result. ALL returns TRUE only if the comparison is TRUE for every value in the result.

ANY — Employees earning more than ANY junior employee

T-SQL
SELECT EmpName, Salary
FROM   Employees
WHERE  Salary > ANY (
           SELECT Salary
           FROM   Employees
           WHERE  JobLevel = 'Junior'
       );

-- Equivalent to: Salary > MIN(junior salaries)
-- Returns all employees who earn more than the lowest-paid junior

ALL — Employees earning more than ALL junior employees

T-SQL
SELECT EmpName, Salary
FROM   Employees
WHERE  Salary > ALL (
           SELECT Salary
           FROM   Employees
           WHERE  JobLevel = 'Junior'
       );

-- Equivalent to: Salary > MAX(junior salaries)
-- Returns employees who earn more than every single junior

15Subquery Inside SELECT

A subquery in the SELECT list must return a single scalar value (one row, one column) for each row of the outer query. This is called a scalar subquery in the projection.

Example — Show employee salary as % of department total

T-SQL
SELECT
    e.EmpName,
    e.Salary,
    (SELECT SUM(e2.Salary)
     FROM   Employees e2
     WHERE  e2.DeptID = e.DeptID)       AS DeptTotalSalary,
    CAST(e.Salary * 100.0 /
        (SELECT SUM(e2.Salary)
         FROM   Employees e2
         WHERE  e2.DeptID = e.DeptID) AS DECIMAL(5,2)) AS PctOfDeptSalary
FROM   Employees e;

-- Better approach for production: use window functions
-- SUM(Salary) OVER (PARTITION BY DeptID) — computes in one pass

16Subquery Inside FROM — Derived Tables

When a subquery appears in the FROM clause, it is called a derived table. It acts like a temporary, inline view. Derived tables must be given an alias.

Example — Average salary by department, then filter high-paid departments

T-SQL
SELECT
    dept_stats.DeptID,
    dept_stats.AvgSalary,
    dept_stats.HeadCount
FROM (
    SELECT
        DeptID,
        AVG(Salary)   AS AvgSalary,
        COUNT(*)     AS HeadCount
    FROM Employees
    GROUP BY DeptID
) AS dept_stats     -- alias is REQUIRED
WHERE dept_stats.AvgSalary > 60000;

-- This two-step approach is common:
-- Step 1 (inner): Aggregate per department
-- Step 2 (outer): Filter on the aggregated result
-- You CANNOT use a WHERE clause on an aggregate directly without HAVING
ℹ️
Derived Table vs CTE

Derived tables are inline and cannot be referenced more than once in the same query. If you need to reference the same subquery twice, use a CTE (WITH clause) instead — it is cleaner and reusable within the query.

17Subquery in WHERE & HAVING

WHERE — Filter rows with a subquery

T-SQL
-- Latest order for each customer (using MAX subquery)
SELECT o.CustomerID, o.OrderDate, o.Amount
FROM   Orders o
WHERE  o.OrderDate = (
           SELECT MAX(o2.OrderDate)
           FROM   Orders o2
           WHERE  o2.CustomerID = o.CustomerID
       );

HAVING — Filter groups using a subquery

T-SQL
-- Departments where the total salary bill exceeds
-- the average departmental total
SELECT   DeptID, SUM(Salary) AS TotalSalary
FROM     Employees
GROUP BY DeptID
HAVING   SUM(Salary) > (
             SELECT AVG(dept_total.TotalSal)
             FROM (
                 SELECT SUM(Salary) AS TotalSal
                 FROM   Employees
                 GROUP BY DeptID
             ) AS dept_total
         );

18Subquery in INSERT / UPDATE / DELETE

INSERT with Subquery — Archive old orders

T-SQL
INSERT INTO Orders_Archive (OrderID, CustomerID, Amount, OrderDate)
SELECT OrderID, CustomerID, Amount, OrderDate
FROM   Orders
WHERE  OrderDate < '2022-01-01';

-- Copies all orders older than 2022 into the archive table

UPDATE with Subquery — Apply loyalty discount

T-SQL
UPDATE Orders
SET    Amount = Amount * 0.90   -- 10% discount
WHERE  CustomerID IN (
           SELECT CustomerID
           FROM   Customers
           WHERE  LoyaltyTier = 'Gold'
       );

-- Updates order amounts for all Gold tier customers

DELETE with Subquery — Remove cancelled orders

T-SQL
DELETE FROM Orders
WHERE  CustomerID IN (
           SELECT CustomerID
           FROM   Customers
           WHERE  Status = 'Cancelled'
       );

-- Deletes orders belonging to cancelled customer accounts
-- ALWAYS run as SELECT first to verify affected rows!
🚨
Safety First on DELETE/UPDATE

Before executing a DELETE or UPDATE with a subquery, always convert it to a SELECT first and verify the affected rows. Wrap in a transaction with BEGIN TRANSACTION and ROLLBACK if uncertain.

19Subquery vs JOIN — Comparison

Both subqueries and JOINs can solve the same problems, but they differ in readability, performance, and flexibility.

When to Use Each

  • Subquery: Simple, one-time filters or scalar values. Short queries where inline context is clear.
  • CTE: Complex queries with multiple logical steps, recursion (hierarchy trees), or when the same subquery is needed multiple times in one query.
  • Temp Table: Large intermediate result sets that benefit from indexing, or when the result needs to be used across multiple separate statements.

21Performance Considerations

Execution Plan Analysis

Always check the execution plan with SET STATISTICS IO ON and INCLUDE ACTUAL EXECUTION PLAN in SSMS. Look for:

  • Nested Loop Join: May indicate correlated subquery inefficiency
  • Table Scans vs Index Seeks: Scans on large tables are expensive
  • Estimated vs Actual rows: Large discrepancy means stale statistics
  • Spool operations: Often introduced for correlated subqueries

SARGability — Making Subquery Filters Index-Friendly

A condition is SARGable (Search ARGument Able) if SQL Server can use an index seek rather than a full scan. Subquery conditions must also be SARGable to be efficient.

T-SQL — Non-SARGable vs SARGable
-- ❌ Non-SARGable (function wraps the column → full scan)
WHERE YEAR(OrderDate) = 2024

-- ✔ SARGable (range condition → index seek)
WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'

-- ❌ Non-SARGable (function on column)
WHERE UPPER(EmpName) = 'ALICE'

-- ✔ SARGable (apply function to the literal)
WHERE EmpName = 'Alice' -- with a case-insensitive collation

Correlated Subquery → Window Function Optimization

T-SQL — Before & After Optimization
-- ❌ BEFORE: Correlated subquery — O(n²) on large tables
SELECT e.EmpName, e.Salary
FROM   Employees e
WHERE  e.Salary > (
           SELECT AVG(e2.Salary) FROM Employees e2
           WHERE e2.DeptID = e.DeptID
       );

-- ✔ AFTER: Window function — single table pass O(n)
SELECT EmpName, Salary
FROM (
    SELECT
        EmpName, Salary,
        AVG(Salary) OVER (PARTITION BY DeptID) AS DeptAvg
    FROM Employees
) ranked
WHERE Salary > DeptAvg;

22Common Mistakes & NULL Traps

Mistake 1 — NOT IN with NULLs (Silent Bug)

T-SQL
-- ❌ BUGGY: If Orders.CustomerID has any NULL, returns 0 rows!
SELECT CustomerName FROM Customers
WHERE  CustomerID NOT IN (SELECT CustomerID FROM Orders);

-- ✔ SAFE: Use NOT EXISTS to handle NULLs correctly
SELECT CustomerName FROM Customers c
WHERE  NOT EXISTS (
           SELECT 1 FROM Orders o
           WHERE  o.CustomerID = c.CustomerID
       );

Mistake 2 — Single-row subquery returning multiple rows

T-SQL
-- ❌ ERROR if multiple employees share the max salary
SELECT EmpName FROM Employees
WHERE  Salary = (SELECT Salary FROM Employees WHERE DeptID = 10);

-- ✔ SAFE: Use aggregation to guarantee scalar
SELECT EmpName FROM Employees
WHERE  Salary = (SELECT MAX(Salary) FROM Employees WHERE DeptID = 10);

Mistake 3 — Forgetting the alias on a derived table

T-SQL
-- ❌ ERROR: Every derived table must have an alias
SELECT * FROM (SELECT EmpName, Salary FROM Employees);

-- ✔ CORRECT
SELECT * FROM (SELECT EmpName, Salary FROM Employees) AS emp_data;

Mistake 4 — Using ORDER BY inside a subquery (invalid in SQL Server)

T-SQL
-- ❌ ERROR: ORDER BY not allowed in subqueries without TOP/OFFSET
SELECT EmpName FROM Employees
WHERE  DeptID IN (SELECT DeptID FROM Departments ORDER BY Budget);

-- ✔ CORRECT: Add TOP to allow ORDER BY
WHERE  DeptID IN (SELECT TOP 100 PERCENT DeptID FROM Departments ORDER BY Budget);

23Real-World Business Scenarios

Scenario 1 — HR: Employees above their department average salary

T-SQL
SELECT
    e.EmpName, e.Salary, d.DeptName,
    dept.AvgSalary,
    e.Salary - dept.AvgSalary AS AboveAvgBy
FROM      Employees    e
JOIN      Departments  d    ON d.DeptID = e.DeptID
JOIN (
    SELECT DeptID, AVG(Salary) AS AvgSalary
    FROM   Employees
    GROUP BY DeptID
) dept ON dept.DeptID = e.DeptID
WHERE e.Salary > dept.AvgSalary
ORDER BY AboveAvgBy DESC;

Scenario 2 — Marketing: Customers who never placed an order

T-SQL
SELECT c.CustomerName, c.Email, c.City
FROM   Customers c
WHERE  NOT EXISTS (
           SELECT 1 FROM Orders o
           WHERE  o.CustomerID = c.CustomerID
       )
ORDER BY c.CustomerName;

-- Use case: Re-engagement email campaign targeting inactive registrants

Scenario 3 — Sales: Top 5 selling products

T-SQL
SELECT p.ProductName, p.Category, sales.TotalRevenue
FROM   Products p
JOIN (
    SELECT   ProductID, SUM(Amount) AS TotalRevenue
    FROM     Orders
    GROUP BY ProductID
) sales ON sales.ProductID = p.ProductID
ORDER BY sales.TotalRevenue DESC
OFFSET 0 ROWS FETCH NEXT 5 ROWS ONLY;

Scenario 4 — Data Quality: Detecting duplicate records

T-SQL
SELECT CustomerID, Email, COUNT(*) AS DuplicateCount
FROM   Customers
WHERE  Email IN (
           SELECT   Email
           FROM     Customers
           GROUP BY Email
           HAVING   COUNT(*) > 1
       )
GROUP BY CustomerID, Email
ORDER BY Email;

Scenario 5 — Finance: Latest transaction per account

T-SQL
SELECT t.AccountID, t.TransactionDate, t.Amount, t.TransactionType
FROM   Transactions t
WHERE  t.TransactionDate = (
           SELECT MAX(t2.TransactionDate)
           FROM   Transactions t2
           WHERE  t2.AccountID = t.AccountID
       )
ORDER BY t.AccountID;

Scenario 6 — Fraud Detection: Customers with unusually large orders

T-SQL
SELECT o.OrderID, o.CustomerID, o.Amount, c.CustomerName
FROM   Orders    o
JOIN   Customers c ON c.CustomerID = o.CustomerID
WHERE  o.Amount > (
           SELECT AVG(Amount) + 3 * STDEV(Amount)
           FROM   Orders
       )
ORDER BY o.Amount DESC;

-- Flags orders more than 3 standard deviations above the mean
-- Classic statistical anomaly detection for fraud review

Scenario 7 — Inventory: Products below reorder threshold

T-SQL
SELECT p.ProductName, p.Stock, p.ReorderLevel,
       (SELECT SUM(o.Quantity)
        FROM   Orders o
        WHERE  o.ProductID = p.ProductID
          AND  o.OrderDate >= DATEADD(DAY, -30, GETDATE())) AS Last30DaySales
FROM   Products p
WHERE  p.Stock < p.ReorderLevel;

Scenario 8 — Financial Reporting: Department budget utilization

T-SQL
SELECT
    d.DeptName,
    d.Budget,
    dept_spend.TotalSalary,
    CAST(dept_spend.TotalSalary * 100.0 / d.Budget AS DECIMAL(5,2)) AS BudgetUsedPct
FROM   Departments d
JOIN (
    SELECT   DeptID, SUM(Salary) AS TotalSalary
    FROM     Employees
    GROUP BY DeptID
) dept_spend ON dept_spend.DeptID = d.DeptID
ORDER BY BudgetUsedPct DESC;

24Interview Questions & Answers

🟢 Beginner Level

Q1 — What is a subquery?

Answer: A subquery is a SELECT statement nested inside another SQL statement. The inner query runs first and its result is passed to the outer query. Subqueries can appear in SELECT, FROM, WHERE, or HAVING clauses.

Q2 — What is the difference between a correlated and non-correlated subquery?

Answer: A non-correlated subquery is independent of the outer query — it runs once, and the result is reused. A correlated subquery references columns from the outer query, so it must re-execute for every row the outer query processes. Correlated subqueries are more powerful but potentially much slower on large data.

Q3 — Can a subquery return multiple columns?

Answer: Yes — but only when used in specific contexts. In a FROM clause (derived table), a subquery can return multiple columns. In a WHERE clause with comparison operators like =, it must return exactly one column. With EXISTS, the columns returned are irrelevant since only row existence is checked.

🟡 Intermediate Level

Q4 — Why does NOT IN with a subquery sometimes return 0 rows unexpectedly?

Answer: Because of NULLs. If the subquery returns any NULL value, every row in the outer query evaluates to UNKNOWN (not TRUE or FALSE) when compared with NOT IN, so no rows pass the filter. The fix is to add WHERE column IS NOT NULL inside the subquery, or replace NOT IN with NOT EXISTS.

Q5 — What is the difference between EXISTS and IN?

Answer: EXISTS checks for the presence of any matching row and stops as soon as it finds one (short-circuit). IN builds the complete result set first, then checks membership. EXISTS is generally preferred for large datasets or when the subquery may return many rows. EXISTS also handles NULLs safely, while NOT IN does not.

Q6 — What is a derived table?

Answer: A derived table is a subquery placed in the FROM clause and given an alias. It acts as an inline view — the result set of the subquery is treated as a virtual table for the outer query. Every derived table must have an alias. Example: SELECT * FROM (SELECT DeptID, AVG(Salary) AS AvgSal FROM Employees GROUP BY DeptID) AS dept_stats.

🔴 Advanced Level

Q7 — When would you choose a CTE over a subquery?

Answer: Use a CTE when: (1) the same subquery result is needed more than once in the query; (2) the query has multiple logical steps that benefit from being named; (3) recursion is needed (hierarchical data like org charts, bill of materials); (4) readability is a priority for long-term maintenance. Subqueries are fine for simple, one-time inline logic.

Q8 — How would you optimize a correlated subquery that runs slowly?

Answer: Several approaches: (1) Replace with a window function (AVG() OVER (PARTITION BY ...)) which does a single table scan; (2) Rewrite as a JOIN with a derived table or CTE that pre-aggregates; (3) Add appropriate indexes on the correlated column and the aggregate column; (4) Check the execution plan for nested loop operations which indicate row-by-row processing.

Q9 — Tricky: Write a query to find the second-highest salary using a subquery
T-SQL
-- Method 1: Subquery
SELECT MAX(Salary) AS SecondHighest
FROM   Employees
WHERE  Salary < (SELECT MAX(Salary) FROM Employees);

-- Method 2: Window function (more flexible)
SELECT Salary
FROM (
    SELECT Salary,
           DENSE_RANK() OVER (ORDER BY Salary DESC) AS Rnk
    FROM   Employees
) ranked
WHERE Rnk = 2;
Q10 — Scenario: You have a query using NOT IN that was working, then suddenly returns 0 rows after new data was inserted. What is likely wrong?

Answer: A NULL value was inserted into the column referenced by the subquery in NOT IN. When a NULL appears in the subquery result, SQL Server cannot determine if any outer value is "not equal to NULL" (since NULL ≠ NULL is UNKNOWN), so the entire result collapses to zero rows. Fix: add WHERE column IS NOT NULL in the subquery or convert to NOT EXISTS.
