01 What is a CTE?

A Common Table Expression (CTE) is a named, temporary result set that you define within a single SQL statement. Think of it as giving a name to a complex query so you can refer to it simply, like a variable for a result set — but without creating any permanent database object.

ℹ️
Definition

A CTE provides a way to write auxiliary statements for use in a larger SELECT, INSERT, UPDATE, DELETE, or MERGE statement. It exists only for the duration of a single query execution.

Introduced in SQL Server 2005 as part of the ANSI SQL:1999 standard, CTEs have become one of the most powerful and readable features in T-SQL. They are especially valued for recursive operations like organizational hierarchies and bill-of-materials processing.

02 Why Use CTEs?

CTEs solve several real problems that arise when writing complex SQL:

🧹 Readability

Break one massive query into named, logical steps. Future you (and teammates) will thank you.

🔁 Reusability

Reference the same CTE multiple times in the outer query instead of repeating the subquery.

🌳 Recursion

Only CTEs (not subqueries or views) support recursive self-reference — essential for hierarchical data.

⚡ No Persistence

Zero schema pollution. No temp tables or views left behind — the CTE evaporates after the statement.

🛠️ DML Friendly

Use CTEs as the source for INSERT, UPDATE, DELETE, and MERGE statements.

📐 Structured Thinking

Forces you to decompose the problem into clear stages, improving both development and debugging.

03 How CTEs Work Internally

Understanding internals helps you write efficient CTEs and avoid surprises.

⚠️
Common Misconception

CTEs are not materialized (pre-computed and cached) by default in SQL Server. The optimizer may inline the CTE definition wherever it is referenced — just like a subquery. This means a CTE referenced twice may execute twice.

Execution Flow

Step 1: Parser reads the WITH clause and registers the CTE name.
↓
Step 2: Query optimizer receives the full statement (CTE + main query).
↓
Step 3: Optimizer may inline the CTE definition or spool it depending on cost.
↓
Step 4: Execution plan is generated and executed.
↓
Step 5: Results returned. CTE scope ends immediately.

Spool vs. Inline

BehaviorWhen It HappensEffect
InlinedDefault — optimizer merges CTE into main queryNo extra cost; CTE is "transparent"
Spooled (Row/Table Spool)Recursive CTE or optimizer chooses it for join reuseResults stored in tempdb worktable
Recursive AnchorFirst part of a recursive CTEExecuted once; seeds the result set
Recursive RecursionSecond part references CTE itselfRepeated until no rows returned
💡
Pro Tip

Use SET STATISTICS IO ON and check execution plans with Ctrl+M in SSMS to see exactly how SQL Server handles your CTE — whether it spools or inlines.

04 CTE Syntax Breakdown

Basic Syntax

WITH CTE_Name (Column1, Column2, ...) -- optional column list
AS (
    -- CTE query definition
    SELECT ...
    FROM ...
    WHERE ...
)
-- Main query that uses the CTE
SELECT * FROM CTE_Name;

Syntax Rules

  • The WITH keyword must be the first word of the statement, OR it must be preceded by a semicolon if another statement appears before it.
  • The CTE name must be unique within the same WITH clause.
  • The column alias list is optional unless the inner query uses expressions without explicit aliases.
  • The final outer query (SELECT, INSERT, UPDATE, DELETE, MERGE) must immediately follow the CTE block with no intervening statements.
  • You cannot define indexes, defaults, or constraints on a CTE.
  • A CTE cannot reference itself unless it is a recursive CTE.
🚨
Important

Always use a semicolon (;) to terminate the statement before a WITH clause. If you don't, SQL Server may interpret WITH as part of the previous statement.

Semicolon Best Practice

T-SQL
-- ✔ Safe approach: always precede WITH with a semicolon
;WITH MyCTE AS
(
    SELECT EmployeeID, Name, Salary
    FROM Employees
)
SELECT * FROM MyCTE;

;WITH AvgSalary AS
(
    SELECT AVG(Salary) AS AvgPay
    FROM  Employees
)
SELECT e.Name, e.Department, e.Salary
FROM   Employees e
JOIN   AvgSalary a ON e.Salary > a.AvgPay

Line-by-line explanation: The CTE AvgSalary computes a single number — the company-wide average salary ($86,437.50). The outer query joins every employee row against that single-row CTE. Only rows where the employee's salary exceeds the average pass through.

Example 2 – Department Summary

T-SQL
;WITH DeptStats AS
(
    SELECT
        Department,
        COUNT(*)       AS HeadCount,
        AVG(Salary)     AS AvgSalary,
        MAX(Salary)     AS MaxSalary,
        MIN(Salary)     AS MinSalary
    FROM   Employees
    GROUP BY Department
)
SELECT *
FROM   DeptStats
WHERE  HeadCount >= 2
ORDER BY AvgSalary DESC;

The CTE aggregates all employees by department. The outer query then filters only departments with at least 2 people — a filter you cannot easily push into a WHERE clause (it would need HAVING), but with a CTE it reads like plain English.

06 Multiple CTEs

You can chain multiple CTEs in a single WITH clause, separated by commas. Later CTEs can reference earlier ones, allowing step-by-step data pipeline logic.

Syntax

WITH
  CTE_One AS ( ... ),
  CTE_Two AS ( -- can reference CTE_One ... ),
  CTE_Three AS ( -- can reference CTE_One and CTE_Two ... )
SELECT ... FROM CTE_Three ...;

Business Example – Sales Pipeline Report

T-SQL
;WITH
-- Step 1: Summarize sales per employee per month
MonthlySales AS
(
    SELECT
        EmployeeID,
        FORMAT(SaleDate, 'yyyy-MM') AS SaleMonth,
        SUM(Amount) AS TotalSales
    FROM   Sales
    GROUP BY EmployeeID, FORMAT(SaleDate, 'yyyy-MM')
),
-- Step 2: Rank employees within each month
RankedSales AS
(
    SELECT
        ms.EmployeeID,
        e.Name,
        ms.SaleMonth,
        ms.TotalSales,
        RANK() OVER (PARTITION BY ms.SaleMonth ORDER BY ms.TotalSales DESC) AS SaleRank
    FROM   MonthlySales ms
    JOIN   Employees e ON ms.EmployeeID = e.EmployeeID
)
-- Step 3: Pick top-3 each month
SELECT SaleMonth, Name, TotalSales, SaleRank
FROM   RankedSales
WHERE  SaleRank <= 3
ORDER BY SaleMonth, SaleRank;

Notice how the pipeline reads clearly: aggregate → rank → filter. Each step is self-documenting. Without CTEs, this would be deeply nested subqueries that are very hard to debug.

07 Recursive CTEs

A recursive CTE references itself, allowing it to traverse hierarchical or graph-structured data — trees, org charts, bill of materials, network paths. It is one of the most powerful features of SQL.

Structure

WITH RecursiveCTE AS (
    -- 1. ANCHOR MEMBER: the starting rows (no self-reference)
    SELECT ... FROM BaseTable WHERE <base condition>

    UNION ALL

    -- 2. RECURSIVE MEMBER: self-reference builds the next level
    SELECT ... FROM BaseTable bt
    JOIN RecursiveCTE r ON bt.ParentID = r.ChildID
)
SELECT * FROM RecursiveCTE;

Step-by-Step Execution Visualization

Iteration 0 (Anchor): Execute anchor SELECT → produce initial result set R0
↓
Iteration 1: JOIN BaseTable ON ... WHERE parent IN R0 → produce R1
↓
Iteration 2: JOIN BaseTable ON ... WHERE parent IN R1 → produce R2
↓
... Continue until recursive member returns 0 rows
↓
Final Result: UNION ALL of R0 ∪ R1 ∪ R2 ∪ ... ∪ Rn
💎
Key Rule

The recursive member must use UNION ALL (not UNION, INTERSECT, or EXCEPT). The anchor and recursive parts must have the same number of columns with compatible data types. Aggregates (GROUP BY, HAVING, SUM, etc.) are not allowed in the recursive member.

08 Employee-Manager Hierarchy

The classic use case: traverse an org chart from CEO down to every individual contributor.

T-SQL · Recursive Hierarchy
;WITH OrgChart AS
(
    -- ANCHOR: top-level employee (no manager)
    SELECT
        EmployeeID,
        Name,
        ManagerID,
        0                        AS Level,
        CAST(Name AS VARCHAR(1000)) AS HierarchyPath
    FROM   Employees
    WHERE  ManagerID IS NULL

    UNION ALL

    -- RECURSIVE: join next level
    SELECT
        e.EmployeeID,
        e.Name,
        e.ManagerID,
        oc.Level + 1,
        CAST(oc.HierarchyPath + ' → ' + e.Name AS VARCHAR(1000))
    FROM   Employees e
    JOIN   OrgChart  oc ON e.ManagerID = oc.EmployeeID
)
SELECT
    REPLICATE('    ', Level) + Name AS OrgView,
    Level,
    HierarchyPath
FROM   OrgChart
ORDER BY HierarchyPath;

09 Category Tree Structure

E-commerce and content management systems frequently store product categories or content folders as self-referencing trees. CTEs traverse these naturally.

T-SQL · Category Tree
CREATE TABLE Categories (
    CategoryID   INT PRIMARY KEY,
    CategoryName VARCHAR(100),
    ParentID     INT NULL   -- NULL = root category
);

INSERT INTO Categories VALUES
(1, 'Electronics',   NULL),
(2, 'Computers',      1),
(3, 'Laptops',        2),
(4, 'Gaming Laptops', 3),
(5, 'Phones',         1),
(6, 'Smartphones',    5),
(7, 'Clothing',       NULL),
(8, 'Men''s',          7);

;WITH CategoryTree AS
(
    SELECT
        CategoryID,
        CategoryName,
        ParentID,
        0                              AS Depth,
        CAST(CategoryName AS NVARCHAR(500)) AS FullPath
    FROM   Categories
    WHERE  ParentID IS NULL

    UNION ALL

    SELECT
        c.CategoryID,
        c.CategoryName,
        c.ParentID,
        ct.Depth + 1,
        CAST(ct.FullPath + ' > ' + c.CategoryName AS NVARCHAR(500))
    FROM   Categories c
    JOIN   CategoryTree ct ON c.ParentID = ct.CategoryID
)
SELECT
    REPLICATE('── ', Depth) + CategoryName AS Tree,
    FullPath
FROM   CategoryTree
ORDER BY FullPath;
💎
Real-World Applications

This same pattern powers: product category navigation, file system browsing, organizational reporting, menu systems, geographic region rollups, and comment thread nesting.

10 Date Series Generation

Generating a complete sequence of dates without gaps is a classic reporting requirement. A recursive CTE can produce one row per day, month, or any interval — without needing a calendar table.

T-SQL · Date Series
-- Generate every day in a 3-month range
;WITH DateSeries AS
(
    SELECT CAST('2024-01-01' AS DATE) AS dt   -- Anchor: start date

    UNION ALL

    SELECT DATEADD(DAY, 1, dt)               -- Recursive: +1 day each step
    FROM   DateSeries
    WHERE  dt < '2024-03-31'                 -- Termination condition
)
SELECT
    dt AS ReportDate,
    DATENAME(WEEKDAY, dt)   AS DayName,
    MONTH(dt)               AS MonthNum
FROM   DateSeries
OPTION (MAXRECURSION 366);  -- Allow up to 366 iterations for full year

Revenue by Day (Left Join to Keep Zero-Sales Days)

T-SQL · Revenue Trend
;WITH DateSeries AS
(
    SELECT CAST('2024-01-01' AS DATE) AS dt
    UNION ALL
    SELECT DATEADD(DAY, 1, dt) FROM DateSeries WHERE dt < '2024-01-31'
)
SELECT
    ds.dt                       AS SaleDate,
    ISNULL(SUM(s.Amount), 0)   AS DailyRevenue
FROM       DateSeries ds
LEFT JOIN  Sales      s  ON  CAST(s.SaleDate AS DATE) = ds.dt
GROUP BY   ds.dt
ORDER BY   ds.dt
OPTION    (MAXRECURSION 100);

Without the date series CTE, days with no sales would be missing from the result. The LEFT JOIN ensures every date appears, with revenue defaulting to 0.

CTE with UPDATE

T-SQL · UPDATE
-- Give Engineering employees a 10% raise
;WITH EngineeringStaff AS
(
    SELECT EmployeeID, Salary
    FROM  Employees
    WHERE Department = 'Engineering'
)
UPDATE EngineeringStaff
SET    Salary = Salary * 1.10;

CTE with DELETE

T-SQL · DELETE (with ROW_NUMBER)
-- Remove duplicate email records, keeping only the most recent one
;WITH Dupes AS
(
    SELECT
        CustomerID,
        Email,
        ROW_NUMBER() OVER (
            PARTITION BY Email
            ORDER BY CreatedAt DESC
        ) AS rn
    FROM Customers
)
DELETE FROM Dupes
WHERE  rn > 1;   -- keep rn=1 (most recent), delete all others
💎
De-duplication Pattern

The CTE + ROW_NUMBER() + DELETE WHERE rn > 1 pattern is the standard, reliable way to remove duplicate rows in SQL Server. It is safer and more readable than NOT IN or NOT EXISTS alternatives.

CTE with MERGE

T-SQL · MERGE (Upsert)
;WITH SourceData AS
(
    SELECT ProductID, ProductName, Price, Stock
    FROM  StageProducts
    WHERE LoadDate = CAST(GETDATE() AS DATE)
)
MERGE Products AS tgt
USING SourceData AS src
    ON tgt.ProductID = src.ProductID
WHEN MATCHED THEN
    UPDATE SET tgt.Price = src.Price, tgt.Stock = src.Stock
WHEN NOT MATCHED BY TARGET THEN
    INSERT (ProductID, ProductName, Price, Stock)
    VALUES (src.ProductID, src.ProductName, src.Price, src.Stock)
WHEN NOT MATCHED BY SOURCE THEN
    UPDATE SET tgt.IsActive = 0;   -- soft-delete removed products

12 CTE vs Subquery vs Temp Table vs View

Feature CTE Subquery Temp Table View
Lifetime Single statement Single statement Session / explicit drop Permanent until dropped
Materialized Usually No No Yes No (unless indexed)
Recursion ✔ Yes ✘ No ✘ No ✘ No
Schema object No No No (session-scoped) Yes
Indexable No No Yes Yes (indexed views)
Reuse across queries No No Yes (same session) Yes (all sessions)
Readability High Low (nested) Medium High
DML source Yes Yes Yes Limited
Best for Complex single query, recursion Simple inline filters Large intermediate data, reuse Shared reusable logic
💡
Decision Guide

Use a CTE when writing a complex single query that benefits from named intermediate steps. Use a Temp Table when the intermediate result set is large (millions of rows) and will be used multiple times or needs indexing. Use a View when the logic is shared across many queries or teams. Use a Subquery only for simple inline filtering.

13 Performance & Optimization

CTE Performance Fundamentals

Because non-recursive CTEs are typically inlined, their performance is usually equivalent to the equivalent subquery. However, there are important nuances:

ScenarioBehaviorRecommendation
CTE referenced onceInlined; optimizer treats it as a subqueryUse CTE freely — no overhead
CTE referenced twice+May execute twice (no automatic caching)Consider #temp table if expensive
Recursive CTE (shallow)Row spooling in tempdb; fast for small hierarchiesWorks well up to ~10,000 rows
Recursive CTE (deep/wide)Spool grows; potential memory pressureUse indexed #temp or HierarchyID type
CTE over large table, no filterFull scans if no predicate pushdownAdd WHERE in the CTE or verify plan

Optimization Example – CTE vs Temp Table

T-SQL · Comparison
-- CTE approach (may re-execute if referenced twice)
;WITH BigCalc AS
(
    SELECT CustomerID, SUM(Amount) TotalSpend
    FROM  Orders  -- 10 million rows
    GROUP BY CustomerID
)
SELECT c.Name, b.TotalSpend
FROM   BigCalc b JOIN Customers c ON b.CustomerID = c.CustomerID
WHERE  b.TotalSpend > 10000;

-- Better for large data — materialize first, then index
SELECT CustomerID, SUM(Amount) TotalSpend
INTO   #BigCalc
FROM   Orders
GROUP BY CustomerID;

CREATE INDEX IX_BigCalc ON #BigCalc (CustomerID);

SELECT c.Name, b.TotalSpend
FROM   #BigCalc b JOIN Customers c ON b.CustomerID = c.CustomerID
WHERE  b.TotalSpend > 10000;

14 MAXRECURSION & Infinite Loops

The MAXRECURSION Option

SQL Server protects against infinite recursion by limiting the default number of recursive iterations to 100. If your recursion exceeds this, you get an error:

🚨
Error Message

The statement terminated. The maximum recursion 100 has been exhausted before statement completion.

T-SQL · MAXRECURSION
;WITH LargeHierarchy AS
(
    SELECT NodeID, ParentID, 0 AS Level FROM Tree WHERE ParentID IS NULL
    UNION ALL
    SELECT t.NodeID, t.ParentID, lh.Level + 1
    FROM   Tree t JOIN LargeHierarchy lh ON t.ParentID = lh.NodeID
)
SELECT * FROM LargeHierarchy
OPTION (MAXRECURSION 500);  -- allow up to 500 levels

-- Use MAXRECURSION 0 to allow UNLIMITED recursion (dangerous!)
OPTION (MAXRECURSION 0);
ValueMeaningUse Case
100 (default)Max 100 recursion levelsMost shallow hierarchies
1–32767Custom limitKnown deep trees, date series
0UnlimitedOnly with guaranteed termination

Preventing Infinite Recursion

Infinite recursion happens when the termination condition is never met — usually caused by circular references in data (e.g., Employee A reports to B, and B reports to A).

T-SQL · Cycle Detection
;WITH SafeHierarchy AS
(
    SELECT
        EmployeeID,
        ManagerID,
        0 AS Level,
        CAST(',' + CAST(EmployeeID AS VARCHAR) + ',' AS VARCHAR(1000)) AS Visited
    FROM   Employees WHERE ManagerID IS NULL

    UNION ALL

    SELECT
        e.EmployeeID,
        e.ManagerID,
        sh.Level + 1,
        sh.Visited + CAST(e.EmployeeID AS VARCHAR) + ','
    FROM   Employees e
    JOIN   SafeHierarchy sh ON e.ManagerID = sh.EmployeeID
    WHERE  sh.Visited NOT LIKE '%,' + CAST(e.EmployeeID AS VARCHAR) + ','  -- cycle check
)
SELECT * FROM SafeHierarchy;

15 Common Mistakes & Best Practices

Common Mistakes

MistakeProblemFix
Missing semicolon before WITH Parse error — SQL Server treats WITH as part of previous statement Always write ;WITH or place a ; on a line by itself before
Referencing CTE outside its query CTE is out of scope — "invalid object name" error Keep CTE and its consuming query as one statement
UNION instead of UNION ALL in recursion Syntax error — SQL Server requires UNION ALL in recursive member Always use UNION ALL between anchor and recursive parts
Aggregate in recursive member Not allowed — GROUP BY, HAVING, SUM etc. are prohibited Move aggregation to the final outer SELECT
Assuming CTE is cached CTE referenced twice executes twice — performance issue Use #temp table when result is expensive and reused
No termination condition in recursive member Infinite loop → MAXRECURSION error Always include a WHERE clause that stops the recursion
ORDER BY inside CTE ORDER BY in a subquery/CTE has no effect (or errors) Place ORDER BY only in the final SELECT
Column count mismatch in recursive parts Error: "The number of columns in the two SELECT statements do not match" Anchor and recursive members must have the same number of columns

Best Practices

Always use ;WITH Name CTEs descriptively Filter early in CTE Set explicit MAXRECURSION Check execution plans Prefer temp tables for large reused sets Test recursion depth Add cycle detection
💡
Naming Convention

Use descriptive CTE names that reflect the business logic — e.g., ActiveCustomersWithOrders, MonthlyRevenueByRegion, OrgChartLevel3. Avoid generic names like CTE1, temp, or data.

16 Advanced Business Scenarios

Scenario 1 – Recursive Bill of Materials (BOM)

Manufacturing systems track how products are made from components, which are themselves made of sub-components — a classic multi-level BOM tree.

T-SQL · BOM
;WITH BOM AS
(
    SELECT
        b.ParentProductID,
        b.ChildProductID,
        p.ProductName,
        b.Quantity,
        1 AS Level,
        b.Quantity AS TotalQty
    FROM   BillOfMaterials b
    JOIN   Products p ON b.ChildProductID = p.ProductID
    WHERE  b.ParentProductID = 1001   -- root product ID

    UNION ALL

    SELECT
        b2.ParentProductID,
        b2.ChildProductID,
        p2.ProductName,
        b2.Quantity,
        bom.Level + 1,
        bom.TotalQty * b2.Quantity   -- cumulative quantity
    FROM   BillOfMaterials b2
    JOIN   Products p2 ON b2.ChildProductID = p2.ProductID
    JOIN   BOM ON b2.ParentProductID = BOM.ChildProductID
)
SELECT
    REPLICATE('  ', Level) + ProductName AS Component,
    Level,
    TotalQty AS RequiredQty
FROM   BOM
ORDER BY Level, ProductName;

Scenario 2 – Financial Ledger Chain

Trace a chain of financial transactions — each transaction references the previous one (e.g., journal entry corrections, linked invoices, or payment chains).

T-SQL · Ledger Chain
;WITH TxChain AS
(
    SELECT
        TxID, Amount, TxDate, PrevTxID,
        Amount AS RunningBalance,
        1 AS ChainStep
    FROM   Transactions
    WHERE  PrevTxID IS NULL   -- origin transaction

    UNION ALL

    SELECT
        t.TxID, t.Amount, t.TxDate, t.PrevTxID,
        tc.RunningBalance + t.Amount,
        tc.ChainStep + 1
    FROM   Transactions t
    JOIN   TxChain tc ON t.PrevTxID = tc.TxID
)
SELECT ChainStep, TxDate, Amount, RunningBalance
FROM   TxChain
ORDER BY ChainStep;

Scenario 3 – Social Network Relationship Traversal

Find all connections reachable from a given user within N degrees (e.g., LinkedIn "2nd degree connections").

T-SQL · Social Graph
;WITH Network AS
(
    SELECT
        UserID, ConnectedUserID, 1 AS Degree,
        CAST(UserID AS VARCHAR(100)) + '->' +
          CAST(ConnectedUserID AS VARCHAR(100)) AS Path
    FROM   Connections
    WHERE  UserID = 42   -- starting user

    UNION ALL

    SELECT
        c.UserID, c.ConnectedUserID, n.Degree + 1,
        n.Path + '->' + CAST(c.ConnectedUserID AS VARCHAR(100))
    FROM   Connections c
    JOIN   Network n ON c.UserID = n.ConnectedUserID
    WHERE  n.Degree < 3   -- stop at 3rd degree
      AND  n.Path NOT LIKE '%' + CAST(c.ConnectedUserID AS VARCHAR) + '%'
)
SELECT DISTINCT ConnectedUserID, Degree, Path FROM Network ORDER BY Degree;

17 Interview Questions & Answers

Beginner Level

Q1 · What is a CTE and how is it different from a subquery?

A CTE (Common Table Expression) is a named temporary result set defined with the WITH keyword before a query. Like a subquery, it exists only for the duration of one statement, but differs in key ways: it has a name (reusable within the same query), it can be recursive, and it greatly improves readability. A subquery is anonymous and embedded inline; a CTE sits at the top, named and readable.

Q2 · Can a CTE be used with INSERT, UPDATE, and DELETE?

Yes. You can write a CTE that defines which rows to modify, and then issue an UPDATE or DELETE directly against the CTE. For example, the classic de-duplication pattern uses ROW_NUMBER() inside a CTE and then DELETE WHERE rn > 1 to remove duplicates cleanly.

Q3 · Can you define multiple CTEs in one statement?

Yes. You use a single WITH keyword and separate each CTE with a comma. Later CTEs can reference earlier ones. This enables a step-by-step data pipeline within a single SQL statement — for example: aggregate → rank → filter.

Q4 · Is a CTE materialized (stored in memory or disk)?

By default, non-recursive CTEs are not materialized. The SQL Server optimizer typically inlines the CTE definition wherever it is referenced, similar to how a subquery works. However, recursive CTEs use an internal spool (row buffer) in tempdb during execution. If you need guaranteed materialization, use a #temp table instead.

Intermediate Level

Q5 · Why must you use UNION ALL (not UNION) in a recursive CTE?

SQL Server's recursive CTE syntax requires UNION ALL between the anchor and recursive members. UNION (which removes duplicates) is not permitted because: (1) duplicate elimination inside the recursion would require sorting and comparing all previous rows at every iteration — too expensive; (2) duplicates in recursive hierarchies are often meaningful (same node reachable via different paths). If you need distinct final results, apply DISTINCT or GROUP BY in the outer SELECT.

Q6 · What is the default MAXRECURSION value and when would you change it?

The default is 100. You would increase it when processing hierarchies deeper than 100 levels (e.g., large org charts, deep BOM structures), or when generating date/number series longer than 100 rows. Use OPTION (MAXRECURSION 0) only when you are absolutely certain the recursion terminates — for example, iterating over a fixed list. Never use 0 on untrusted data without cycle detection.

Q7 · When would you choose a temp table over a CTE?

Choose a #temp table when: (1) the intermediate result set is very large (millions of rows) and referenced more than once — CTEs may re-execute; (2) you need to add an index to the intermediate data to speed up subsequent joins; (3) you need to reuse the result across multiple separate statements in the same session.

Advanced Level

Q8 · How do you detect and prevent circular references in a recursive CTE?

The most common technique is to build a path string (e.g., ',1,3,7,') as you recurse, and in the recursive member's WHERE clause check that the next node ID is not already in the path using NOT LIKE '%,nodeID,%'. This stops the recursion if a cycle is detected. Alternatively, add a Level < MaxDepth guard as a safety net. In production systems, data integrity constraints (e.g., preventing self-referential parent IDs) are the first line of defense.

Q9 · Explain how you would use a CTE to de-duplicate rows, preserving only the most recent record per group.

Use ROW_NUMBER() OVER (PARTITION BY <group key> ORDER BY <tie-breaker DESC>) inside the CTE. This assigns rank 1 to the row you want to keep (most recent, based on a date or ID). Then in the outer statement, DELETE FROM the_cte WHERE rn > 1. Because the CTE is updatable (it maps 1:1 to the base table), the delete propagates correctly to the underlying rows.

Q10 · Scenario: You have a 50-level deep product hierarchy (BOM) and your recursive CTE throws a MAXRECURSION error. Walk through your solution.

Step 1: Determine the actual depth — query the data to find the true max depth (e.g., with a preliminary recursive CTE using OPTION (MAXRECURSION 0) with a Level < 100 guard).
Step 2: Add OPTION (MAXRECURSION 500) (or the appropriate value) to the query hint.
Step 3: Add a cycle-detection mechanism in the recursive member's WHERE clause to protect against circular BOM data.
Step 4: For very wide or deep BOMs (10,000+ nodes), consider pre-flattening the hierarchy into a closure table or using the HierarchyID data type for O(log n) path operations.

Q11 · How does a CTE compare to an indexed view in terms of performance?

An indexed view (materialized view) stores its result set on disk with a clustered index, making queries against it extremely fast for read-heavy workloads. A CTE is computed on demand every time the query runs — it is never persisted. For reports that run frequently on expensive aggregations over large tables, an indexed view or a pre-aggregated summary table will outperform a CTE significantly. Use CTEs for query readability and single-use logic; use indexed views for frequently-queried, stable aggregations.

19 Summary

📌 What CTEs Are

Named temporary result sets defined with WITH ... AS (...) that exist for exactly one statement's lifetime. Introduced in SQL Server 2005, part of ANSI SQL:1999.

🔑 Core Strengths

Readability, recursion support, DML compatibility, zero schema pollution, and the ability to chain multiple named steps into one readable query.

🌳 Recursive Power

Anchor + UNION ALL + recursive member. Traverses hierarchies, trees, graphs, and chains that are impossible with plain subqueries or views.

⚙️ Performance Reality

Not automatically materialized. Referenced twice = executed twice. For large, reused results: use #temp tables with indexes. Check execution plans always.

🛡️ Safety Rules

Set explicit MAXRECURSION. Add cycle-detection to recursive queries on untrusted data. Always use ;WITH to avoid parse errors.

🏢 Business Value

Powers org charts, BOM calculations, financial ledger chains, date series reporting, network traversal, category trees, and de-duplication pipelines.

💎
Final Advice from the Author

Mastering CTEs will transform how you write SQL. Start by replacing your most deeply nested subqueries with named CTEs. Once comfortable, explore recursive CTEs on real hierarchical data — nothing else in SQL is as elegant for that class of problem. Always profile your queries: a well-written CTE that maps poorly to the execution engine should be refactored, not blindly trusted. The best SQL developer is not one who knows the most syntax, but one who understands what happens under the hood.
