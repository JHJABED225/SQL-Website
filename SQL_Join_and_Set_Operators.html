01Introduction to Joins

Joins combine rows from two or more tables based on a relatable column. Choosing the right type of join is important because it determines which rows to keep. You can join on the basis of more than one column.

Core Joins

INNER · LEFT · RIGHT · FULL

Advanced Joins

CROSS · ANTI JOIN (NOT EXISTS / LEFT JOIN WHERE NULL)

ℹ️
Key Concept

The keyword JOIN alone (without a qualifier) defaults to INNER JOIN in SQL Server. Both are functionally identical.

02INNER JOIN

Inner join returns matching records only. Only rows where the join condition is satisfied in both tables appear in the result set.

📌
Classic Example

Retrieve all customers along with their orders, but only for customers who have actually placed an order.

Task 01

Details about orders along with customer details.

T-SQL
SELECT C.CustomerID,
       TRIM(C.FirstName) + ' ' + TRIM(C.LastName) AS FullName,
       C.Country, C.Score,
       O.OrderID, O.OrderDate, O.Sales
FROM  sales.customers C
INNER JOIN sales.orders AS O
  ON C.CustomerID = O.CustomerID
ORDER BY C.CustomerID
Task 02

Details about only employees who have helped in placing orders.

T-SQL
SELECT *
FROM  sales.employees e
JOIN  sales.orders AS o  -- JOIN = INNER JOIN
  ON e.employeeID = o.SalesPersonID
ORDER BY e.employeeID

Use Cases

  • Get employees with valid departments.
  • Find orders with customers.
  • Data consistency check (only for matched records).

03LEFT JOIN

Left join returns all rows from the left table and matching rows from the right table. Unmatched right-side rows return NULL.

Use Cases

  • Find employees without departments.
  • Show all customers along with their orders (if any).
  • Reporting including missing relationships.
Task 01

Details about only customers who have placed orders (using LEFT JOIN + NULL filter).

T-SQL
SELECT C.CustomerID,
       TRIM(C.FirstName) + ' ' + TRIM(C.LastName) AS FullName,
       C.Country, C.Score,
       O.OrderID, O.OrderDate, O.Sales
FROM  sales.customers C
LEFT JOIN sales.orders AS O
  ON C.CustomerID = O.CustomerID
WHERE O.OrderID IS NOT NULL
ORDER BY C.CustomerID
Task 02

Details about all employees along with order-placing information (including those with no orders).

T-SQL
SELECT *
FROM  sales.employees e
LEFT JOIN sales.orders AS o
  ON e.employeeID = o.SalesPersonID
ORDER BY e.employeeID
💡
Pro Tip

Adding WHERE right_table.key IS NULL after a LEFT JOIN effectively turns it into an Anti Join — returning only rows with no match in the right table.

04RIGHT JOIN

Right join returns all rows from the right table and matching rows from the left. Unmatched left-side rows return NULL.

The task of a RIGHT JOIN can always be accomplished with a LEFT JOIN by switching the order of tables in the query.

Task

Details about all customers along with their orders (if they placed orders).

T-SQL
SELECT c.CustomerID,
       TRIM(c.FirstName) + ' ' + TRIM(c.LastName) AS FullName,
       c.Country, c.Score,
       o.OrderID, o.OrderDate, o.Sales
FROM  sales.orders AS o
RIGHT JOIN sales.customers AS c
  ON o.CustomerID = c.CustomerID
ORDER BY c.CustomerID

Use Cases

  • Find departments with no employees.
  • Ensure all references / master data is shown.
  • Reverse logic of LEFT JOIN (when the "all rows" table is the second table in FROM).

05FULL JOIN

Full join (also called FULL OUTER JOIN) returns all rows from both tables. Where no match exists on either side, NULL is returned for the missing columns.

Task

All customers and all orders — matched or not.

T-SQL
SELECT *
FROM  sales.orders AS o
FULL JOIN sales.customers AS c
  ON o.CustomerID = c.CustomerID
⚠️
Watch Out

FULL JOIN can produce large result sets with many NULL values. Use WHERE filters or COALESCE to manage NULLs in the output.

06CROSS JOIN

CROSS JOIN returns all combinations of two tables — the Cartesian product. If Table A has m rows and Table B has n rows, the result has m × n rows.

Use Cases

  • Generate all possible combinations (e.g., product × region).
  • Create test / seed data.
  • Matrix and report building.
Task

Find all combinations of customer and employee information.

T-SQL
SELECT *
FROM  sales.customers
CROSS JOIN sales.employees
🚨
Danger — Performance Risk

CROSS JOIN on large tables can produce millions of rows and overwhelm the server. Always test on a small subset or add a WHERE clause to limit output.

07ANTI JOIN

An Anti Join returns rows in Table 1 that do not exist in Table 2. SQL Server has no direct ANTI JOIN syntax — it is implemented via two common approaches.

ℹ️
Use Case

Return rows in table 1 that do not exist in table 2. Great for finding orphaned records, inactive users, or data gaps.

Method 1 — LEFT JOIN + NULL Check

Task

Find customer details with no orders.

T-SQL
SELECT C.CustomerID, C.FirstName, C.LastName,
       C.Country, C.Score
FROM  sales.customers AS C
LEFT JOIN sales.orders AS O
  ON C.CustomerID = O.CustomerID
WHERE O.OrderID IS NULL

Method 2 — NOT EXISTS (Best Practice)

Task

Find customer details with no orders.

T-SQL
SELECT *
FROM  sales.customers AS C
WHERE NOT EXISTS (
    SELECT 1
    FROM  sales.orders AS O
    WHERE C.CustomerID = O.CustomerID
)
✅
Best Practice

NOT EXISTS is generally preferred over LEFT JOIN + NULL because it short-circuits on the first match, can be more efficient with the query optimizer, and expresses intent more clearly.

08Joining 3+ Tables

Used when data is spread across multiple tables. Each additional JOIN clause links a new table to the existing result set.

Task

Join 3 tables to retrieve important information about products, orders, and customers.

T-SQL
SELECT o.OrderID,
       p.Product,
       o.Sales,
       c.FirstName,
       c.LastName,
       p.Category,
       o.OrderDate
FROM  sales.orders AS o
JOIN  sales.customers AS c  ON o.CustomerID = c.CustomerID
JOIN  sales.products  AS p  ON o.ProductID  = p.ProductID

Use Cases

  • Reporting dashboards requiring data from multiple entities.
  • Data warehouse queries combining fact and dimension tables.
⚠️
Noticeable Facts
  • LEFT JOIN + WHERE right_col IS NULL = Anti Join.
  • Avoid SELECT * in joins — it reduces performance and pulls unnecessary columns.
  • Proper indexing on join keys is critical for query speed.

09Introduction to Set Operators

Set operators combine the results of two or more SELECT queries into a single result set. Think of them like set theory in mathematics:

A ∪ B
UNION
All unique records from both sets
A ∩ B
INTERSECT
Only records present in both sets
A – B
EXCEPT
Records in A that are not in B
A ∪ B*
UNION ALL
All records including duplicates
📋
Common Rules for All Set Operators
  • Same number of columns in both queries.
  • Same data types (or compatible types) in each corresponding column.
  • Same column order across both queries.
  • Column names come from the first query.

10UNION

Combines results from two queries and removes duplicates. Equivalent to asking "give me all unique records from both tables."

Task 01

Provide all unique cities from both Customers and Suppliers tables.

T-SQL
SELECT City FROM Customers
UNION
SELECT City FROM Suppliers
Task 02

Provide all unique names of customers and employees.

T-SQL
SELECT TRIM(FirstName) AS first_name,
       TRIM(LastName)  AS last_name
FROM  sales.customers

UNION

SELECT TRIM(FirstName),
       TRIM(LastName)
FROM  sales.employees

Use Cases

  • Customer and supplier contact list — combined list of all unique people.
  • Multi-region data merge — combine sales from different regions without duplication.
⚠️
Noticeable Facts
  • Slower than UNION ALL due to duplicate removal.
  • Uses DISTINCT internally, which requires sorting or hashing.

11UNION ALL

Combines results from two queries and keeps all duplicates. No deduplication is performed.

Task

Give all the names of customers and employees (including duplicates).

T-SQL
SELECT TRIM(FirstName) AS first_name,
       TRIM(LastName)  AS last_name
FROM  sales.customers

UNION ALL

SELECT TRIM(FirstName),
       TRIM(LastName)
FROM  sales.employees

Use Cases

  • Transaction logs — you want all records including duplicates.
  • Appending data from multiple sources or time periods.
✅
Performance Note
  • Faster than UNION — no duplicate removal step.
  • No sorting required.
  • Best choice for large datasets where duplicates are acceptable.

12EXCEPT

Returns rows from the first query that are not present in the second query. Order matters — A EXCEPT B ≠ B EXCEPT A.

Task 01

Select all UserIDs that appear in the users table but not in the Logins table.

T-SQL
SELECT userID FROM users
EXCEPT
SELECT UserID FROM Logins
Task 02

Select all names of customers that are not employees.

T-SQL
SELECT TRIM(FirstName) AS first_name,
       TRIM(LastName)  AS last_name
FROM  sales.customers

EXCEPT

SELECT TRIM(FirstName),
       TRIM(LastName)
FROM  sales.employees

Use Cases

  • Inactive users — users registered but never logged in.
  • Missing records / Data validation — expected records not found in a second table.
💡
Noticeable Facts
  • Automatically removes duplicates from the result.
  • Order matters — the first query is the "left" set; position determines what gets excluded.
  • Works like a LEFT ANTI JOIN in join terminology.

13INTERSECT

Returns only common rows that appear in both queries. Equivalent to "give me the matching records only."

Task 01

Select all names of customers that are also employees.

T-SQL
SELECT TRIM(FirstName) AS first_name,
       TRIM(LastName)  AS last_name
FROM  sales.customers

INTERSECT

SELECT TRIM(FirstName),
       TRIM(LastName)
FROM  sales.employees
Task 02

Find customers active in multiple years (loyal customers).

T-SQL
SELECT CustomerID FROM Orders_2023
INTERSECT
SELECT CustomerID FROM Orders_2024

Use Cases

  • Find loyal customers — active across several years.
  • Students who passed all subjects — e.g., students who passed both Math and English.
💎
Noticeable Facts
  • Removes duplicates automatically.
  • Works like an INNER JOIN but uses simpler set logic — no need to specify join keys.
  • Less commonly used than UNION/EXCEPT, but very powerful for overlap analysis.
